## How to Use This Template

### **Purpose of a Glossary**
A glossary serves as a living reference for:
- **Onboarding:** New team members quickly learn domain language
- **Clarity:** Reduces misunderstandings from ambiguous terminology
- **Efficiency:** No need to repeatedly explain common terms
- **Knowledge Retention:** Captures institutional knowledge before expertise leaves
- **Cross-functional Communication:** Bridges terminology gaps between teams

### **When to Add a Term**
Add a term when it:
- **Is used frequently** in your daily work
- **Has specific meaning** in your domain (not general English)
- **Is an acronym** that may not be obvious to others
- **Confuses newcomers** or requires repeated explanation
- **Has evolved** from general use to specific meaning in your context
- **Appears in documentation** and needs definition for readers

### **Term Entry Structure**
Each glossary entry should include:

1. **Term/Acronym:** The word or abbreviation being defined
2. **Definition:** Clear, concise explanation (1-3 sentences)
3. **Example:** Real-world usage to provide context
4. **Related Terms:** Cross-references to similar or connected concepts
5. **Synonyms/Aliases:** Other names for the same concept (if applicable)
6. **See Also:** Links to where this term is used in your documentation

## Example Entries (Customize for Your Domain)

### **Backlog**
A prioritized list of work items, features, or tasks waiting to be completed. In project management, the backlog represents everything that could be done but hasn't been started yet.

**Example:** "We have 47 items in the backlogâ€”let's prioritize the top 10 for next sprint."

**Related Terms:** Sprint, Story Points, Prioritization

---

### **Root Cause Analysis (RCA)**
Systematic process of identifying the fundamental reason a problem occurred, going beyond surface symptoms to address underlying issues.

**Example:** "Instead of just fixing the data error, we did an RCA and discovered the validation script was outdated."

**Common Methods:** 5 Whys, Fishbone Diagram, Fault Tree Analysis  
**See Also:** Project Investigation Template, Problem-Solving Protocols

---

### **SLA (Service Level Agreement)**
Commitment defining the expected level of service, including response times, uptime guarantees, or delivery deadlines. Often used in contracts or internal team agreements.

**Example:** "We have a 24-hour SLA for responding to support tickets."

**Related Terms:** KPI, OLA (Operational Level Agreement), SLO (Service Level Objective)

---

### **Sprint**
Fixed time period (typically 1-4 weeks) during which a team commits to completing specific work items. Common in agile project management methodologies.

**Example:** "Our sprints are 2 weeks long, starting every other Monday."

**Related Terms:** Backlog, Story Points, Sprint Review, Retrospective

---

### **Stakeholder**
Person or group with interest in or influence over a project, decision, or outcome. Stakeholders may include managers, customers, team members, partners, or executives.

**Example:** "I need to send a status update to all project stakeholders by Friday."

**Types:** Internal stakeholders (within organization), External stakeholders (customers, vendors)  
**See Also:** Network Map Template, Communication Standards

---

### **Technical Debt**
Accumulated cost of shortcuts, outdated code, or suboptimal design decisions that make future changes more difficult or time-consuming.

**Example:** "We took on some technical debt to hit the deadlineâ€”now we need to refactor this module."

**Related Concept:** Like financial debt, technical debt accumulates "interest" (increased maintenance cost) over time.

---

### **UAT (User Acceptance Testing)**
Final phase of testing where actual users verify that a system, feature, or deliverable meets their requirements and works as expected in real-world scenarios.

**Example:** "The feature passed QA, but we're waiting for UAT before releasing to production."

**Related Terms:** QA (Quality Assurance), Production, Staging Environment

---

### **Version Control**
System for tracking changes to files over time, allowing multiple people to collaborate and revert to previous versions if needed. Git is the most common version control system.

**Example:** "I committed my changes to version control so the team can see my updates."

**Common Tools:** Git, GitHub, GitLab, Bitbucket, SVN  
**Key Concepts:** Commit, Branch, Merge, Pull Request

## Technical Jargon & Acronyms

*This section captures technical terms, especially acronyms that might not be obvious to newcomers.*

### **API (Application Programming Interface)**
Set of rules and protocols that allow different software applications to communicate with each other.

**Example:** "We're using the OpenAI API to integrate GPT capabilities into our tool."

**Common Context:** Software development, integrations, automation

---

### **CI/CD (Continuous Integration / Continuous Deployment)**
Software development practice where code changes are automatically tested (CI) and deployed to production (CD) frequently.

**Example:** "Our CI/CD pipeline runs tests on every commit and deploys passing builds automatically."

**Related Terms:** DevOps, Automated Testing, Build Pipeline

---

### **ETL (Extract, Transform, Load)**
Data integration process that involves extracting data from sources, transforming it to match target format, and loading it into a destination system.

**Example:** "We run an ETL job nightly to update our analytics database."

**Common Tools:** Apache Airflow, Talend, Informatica, dbt

---

### **CRUD (Create, Read, Update, Delete)**
Four basic operations for managing data in a database or application.

**Example:** "This admin panel provides CRUD functionality for managing user accounts."

**Usage Context:** Database design, API development, application requirements

---

### **OKR (Objectives and Key Results)**
Goal-setting framework where objectives define ambitious goals and key results measure progress toward those objectives.

**Example:** "My Q2 OKR is to improve dashboard load time, with a key result of reducing average load from 8 seconds to 3 seconds."

**Related Terms:** KPI (Key Performance Indicator), Goal Setting

---

## Process & Methodology Terms

*This section captures terms related to how work is organized, executed, and measured.*

### **Retrospective**
Team meeting held at the end of a project or sprint to reflect on what went well, what didn't, and how to improve next time.

**Example:** "In our retrospective, we identified that earlier testing would have prevented last week's issues."

**Common Questions:**
- What went well?
- What could be improved?
- What will we do differently next time?

**See Also:** Sprint, Continuous Improvement, Lessons Learned

---

### **Stand-up / Daily Scrum**
Short daily meeting (typically 15 minutes) where team members share progress, plans, and blockers.

**Example:** "In today's stand-up, I mentioned I'm blocked waiting for API access."

**Typical Structure:**
- What I did yesterday
- What I'm doing today
- Any blockers or help needed

---

### **Definition of Done (DoD)**
Agreed-upon checklist that determines when work is truly complete, ensuring consistent quality standards.

**Example:** "According to our Definition of Done, this feature isn't complete until it has tests, documentation, and passes code review."

**Common DoD Items:**
- Code reviewed and approved
- Tests written and passing
- Documentation updated
- Deployed to staging environment

---

### **Dashboard**
Visual display of key metrics and information, typically updated in real-time or near-real-time, providing at-a-glance view of important data.

**Example:** "I built a dashboard showing customer churn trends by region."

**Common Tools:** Tableau, Power BI, Looker, Grafana

### **KPI (Key Performance Indicator)**
Measurable value that demonstrates effectiveness in achieving key objectives. KPIs track performance against goals.

**Example:** "Our KPI for customer support is to respond to 90% of tickets within 4 hours."

**Related Terms:** Metrics, OKR, Performance Management

---

### **Schema**
Structure that defines how data is organized in a database, including tables, fields, data types, and relationships.

**Example:** "The database schema shows that the 'orders' table links to the 'customers' table via customer_id."

**Common Context:** Database design, data modeling, ETL processes

## Appendix: Glossary Maintenance Guide

### **How to Keep This Glossary Useful**

**Regular Updates:**
1. **Add terms as you encounter them** - Don't wait for formal review
2. **Update definitions** when meaning evolves or you gain deeper understanding
3. **Add examples** from real work to make definitions concrete
4. **Review quarterly** to remove outdated terms or add missing ones

**Quality Standards:**
1. **Define terms simply** - Assume reader is smart but unfamiliar with domain
2. **Use examples** - Real-world context clarifies meaning
3. **Avoid circular definitions** - Don't define terms using other undefined terms
4. **Keep it scannable** - Use formatting (bold, sections) to make terms easy to find
5. **Link related concepts** - Help readers discover connected ideas

**Collaboration:**
1. **Share with new team members** - Use during onboarding
2. **Encourage contributions** - Team members should feel empowered to add terms
3. **Review definitions with subject matter experts** - Ensure accuracy
4. **Update when feedback reveals confusion** - If people ask "what does X mean?", add it

### **When to Split This Glossary**

If your glossary grows beyond ~100 terms, consider splitting into focused glossaries:
- **General Organizational Glossary** - Company-wide terms everyone should know
- **Functional Glossary** - Terms specific to your role (e.g., Data Analytics Glossary, Marketing Glossary)
- **Project Glossary** - Terms specific to a particular project or initiative
- **Technical Glossary** - Deep technical jargon for specialists
